How it works

Three inputs, one beautiful output

Point it at any public GitHub repo. It reads the code, the docs, and the structure — then produces a self-contained explainer page with clear visuals, plain-language sections, and public links.

1

Pick a repo

Paste any public GitHub URL. The pipeline reads the file tree, README, docs, and source code — respecting .gitignore and submodule boundaries.

2

It builds the explainer

The pipeline generates 7 sections: Why built, What problem, Why now, How it works, What solved looks like, How to implement, How to start — plus use cases and a get-started guide.

3

Share publicly

The result is a standalone HTML page. Every link is public — GitHub, npm, releases. No login walls, no gated downloads. Deploy anywhere or host on GitHub Pages.

What you get

Each section opens with a visual before any text. Diagrams, illustrations, and clear typography make complex ideas approachable.

Plain language

No jargon walls. Technical terms are defined inline. Written for someone who has never seen the repo before.

7 structured sections

A proven arc: Why built, What problem, Why now, How it works, What solved looks like, How to implement, How to start.

Use-case gallery

Real scenarios showing how the tool is used. Not abstract features — concrete examples with commands and expected output.

Principles

What makes a good explainer

01

Image before text

Every section opens with a visual. A diagram, a flow, an illustration. Words come second.

02

Human problem first

Start with the pain point a real person has. Then show how the tool fixes it. Features come last.

03

Define terms inline

If a reader might not know a word, explain it right there. No glossary at the bottom nobody reads.

04

Honest limits stated

Every tool has rough edges. Say what they are, plainly, without hedging. Trust is earned by candor.

05

All links public

Every URL goes to something anyone can open without logging in. No gated downloads, no auth walls.

06

Credit the builder

The explainer credits the repo author prominently. If it's not your project, say so on the first screen.
